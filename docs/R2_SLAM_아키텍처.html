전체 구조 개요

이 프로젝트는 RTAB-Map(Real-Time Appearance-Based Mapping)을 주력 SLAM 엔진으로 사용합니다. 로봇이 라이다와 RGB-D 카메라 데이터를 RTAB-Map에 공급하면, RTAB-Map이 스캔 매칭과 루프 클로저를 수행하여 /map 토픽(2D 점유 격자)을 만들어 RViz에 실시간으로 표시합니다.

센서 SLAM 표시/활용 ┌──────────────┐ ┌──────────────────┐ ┌─────────────────────┐ │ RPLIDAR │──>│ │──>│ /map (OccupancyGrid)│──> RViz 2D 지도 │ /scan │ │ │ │ /mapData (3D 클라우드)│──> RViz 포인트클라우드 ├──────────────┤ │ RTAB-Map │ │ /mapGraph (그래프) │──> RViz 그래프 오버레이 │ Orbbec Astra │──>│ rtabmap_slam │ │ map->odom TF │──> 로봇 위치 보정 │ /camera/* │ │ │ └─────────────────────┘ ├──────────────┤ │ │ │ 바퀴 엔코더 │──>│ │ ┌─────────────────────┐ │ /odom │ └──────────────────┘ │ Nav2 (순찰 시) │ └──────────────┘ │ │ map_server + AMCL │ │ │ + DWB planner │ ~/.ros/rtabmap.db └─────────────────────┘ (SLAM 데이터베이스)

1단계: 센서 입력 (laser_bringup)

laser_bringup_launch.py가 로봇 하드웨어와 센서를 동시에 가동합니다.

라이다는 로봇 본체 위에 고정되어 있으므로 base_link → laser 사이의 정적 변환을 퍼블리시합니다. 위치는 x=0.0435m, z=0.11m이며, RPLIDAR 종류(a1/s2 vs 4ROS)에 따라 yaw 방향이 180도 다릅니다.

왜 두 센서를 동시에 쓰나? 라이다는 넓은 시야(360도)로 벽/장애물을 정확히 측정하고, RGB-D 카메라는 시각적 특징(텍스처, 색상)을 제공합니다. RTAB-Map은 이 두 가지를 결합하여 라이다만으로는 구분 못하는 비슷한 복도도 구별할 수 있습니다.

2단계: RTAB-Map SLAM 처리

rtabmap_managed.launch.py(또는 원본 rtabmap_sync_launch.py)가 두 개의 ROS2 노드를 실행합니다.

a) rgbd_sync — 이미지 시간 동기화

RGB 이미지와 Depth 이미지가 약간 다른 시점에 도착하므로, rgbd_sync 노드가 이 둘을 시간 동기화하여 하나의 /rgbd_image 토픽으로 합칩니다.

  • approx_sync: True — 정확히 같은 타임스탬프가 아니어도 0.01초 이내면 매칭
  • 입력: /camera/color/image_raw + /camera/depth/image_raw
  • 출력: /rgbd_image

b) rtabmap — 실제 SLAM 수행

입력 처리 출력 /rgbd_image ──┐ ┌──> /map (2D 점유 격자) ├──> [스캔 매칭] ──> [그래프 최적화] ──┼──> /mapData (3D 포인트클라우드) /scan ────────┤ │ ├──> /mapGraph (노드/링크) │ [루프 클로저 감지] └──> map->odom TF (위치 보정) /odom ────────┘

frame_id: base_footprint — 로봇의 "배꼽" 좌표

한 줄 요약 RTAB-Map에게 "로봇의 위치를 어느 지점 기준으로 계산할지" 알려주는 설정입니다.

로봇은 라이다, 카메라, 바퀴 등 여러 부품으로 이루어져 있고 각 부품마다 자기만의 좌표 원점이 있습니다. 하지만 "로봇이 지금 지도 위 어디에 있는가?"를 말하려면 대표 좌표 하나가 필요합니다. 그게 바로 base_footprint입니다 — 로봇을 위에서 내려다봤을 때 바닥에 비친 그림자(투영)의 중심점이라고 생각하면 됩니다.

  • 왜 base_link(로봇 본체 중심)가 아니라 base_footprint인가? base_link는 로봇 몸체 한가운데(공중에 떠 있는 점일 수도 있음)인 반면, base_footprint는 항상 z=0, 즉 바닥면에 붙어 있습니다. 2D 평면 지도(지도는 위에서 본 평면도)와 좌표를 맞추기에 훨씬 자연스럽습니다.
  • 실생활 비유: 사람의 키를 잴 때 머리 꼭대기가 아니라 발바닥(바닥 접점)을 기준으로 "내가 지금 거실 어디 서 있나"를 말하는 것과 같습니다.
  • 이 값이 잘못되면? 로봇 위치가 지도 위에서 살짝 떠 있거나 기울어진 것처럼 계산되어 장애물 회피나 목표 도달이 부정확해집니다.

Reg/Strategy: 1 — "지금 본 장면이 아까 본 장면과 얼마나 겹치나?"

한 줄 요약 연속된 두 라이다 스캔(또는 두 프레임)을 정렬(맞춰 포개기)하는 방법을 고르는 설정입니다. 값 1은 ICP 방식을 의미합니다.

로봇이 이동하면서 라이다가 매 순간 "주변 점들의 모음"을 새로 찍습니다. SLAM은 이 점들을 이전에 찍은 점들과 비교해서 "내가 얼마나, 어느 방향으로 움직였는지"를 역산합니다. 이 비교/정렬 작업을 스캔 매칭(scan matching)이라 부르고, RTAB-Map은 이를 위한 알고리즘을 몇 가지 제공합니다.

  • ICP (Iterative Closest Point, 값=1): "이전 점들"과 "지금 점들"에서 서로 가장 가까운 점끼리 짝을 짓고, 그 짝들의 거리 합이 최소가 되도록 회전·이동을 반복 계산합니다. 이름 그대로 "가장 가까운 점을 반복해서 찾는" 방식입니다.
  • 왜 ICP를 선택했나? 이 프로젝트는 라이다 스캔 데이터가 풍부하고(360도, 촘촘한 점들), ICP는 점 구름(point cloud) 정렬에 특화되어 있어 라이다 기반 2D 로봇에 적합합니다.
  • 실생활 비유: 어두운 방에서 손으로 벽을 더듬으며 이전에 만졌던 벽의 모양과 지금 만지는 벽의 모양을 비교해 "내가 오른쪽으로 한 발 움직였구나"라고 추측하는 것과 비슷합니다.
  • 실패하면? 정렬에 실패하면(예: 텅 빈 복도처럼 특징 없는 공간) 위치 추정이 흔들리거나 지도가 어긋나게(찢어지게) 보일 수 있습니다.

Reg/Force3DoF: true — "위아래는 무시하고 평면에서만 생각하자"

한 줄 요약 로봇의 위치를 (x, y, 회전각) 3개 값으로만 계산하도록 강제합니다. 즉 "2D 지도"를 만들겠다는 선언입니다.

DoF는 Degree of Freedom(자유도)의 약자로, "물체의 위치/자세를 표현하는 데 필요한 독립적인 숫자의 개수"를 뜻합니다.

  • 3DoF (이 프로젝트): x(앞뒤), y(좌우), yaw(회전각) — 평면 위를 굴러다니는 로봇 그대로
  • 6DoF (드론, 로봇팔 등): 위 3개 + z(높이), roll(좌우 기울임), pitch(앞뒤 기울임) — 3차원 공간을 자유롭게 움직이는 물체

R2는 바퀴로 평평한 바닥을 다니는 로봇이라 하늘로 날아오르거나 옆으로 기울어질 일이 없습니다. 그런데 RGB-D 카메라는 원래 3D 정보(깊이)를 주기 때문에, 그대로 두면 SLAM이 불필요하게 6DoF로 계산하려 들 수 있습니다. Force3DoF: true는 "굳이 z, roll, pitch는 계산하지 말고 0으로 고정해라"라고 강제로 못 박는 설정입니다.

  • 왜 이렇게 하나? 계산량이 줄어들고(빠르고 가벼움), 바닥의 미세한 진동이나 카메라 노이즈 때문에 로봇이 "공중에 살짝 떠 있다"거나 "기울어졌다"고 잘못 추정하는 오류를 원천 차단합니다.
  • 실생활 비유: 종이 위에 그린 평면 미로 지도를 만들 때, "위아래 층"이나 "건물 높이"는 아예 고려하지 않고 오직 가로·세로·방향만 신경 쓰는 것과 같습니다.

RGBD/NeighborLinkRefining: True — "이웃 사진끼리 한 번 더 맞춰보기"

한 줄 요약 SLAM 그래프에서 바로 옆에 있는(시간상 인접한) 두 키프레임 사이의 연결을, 추가로 ICP를 한 번 더 돌려서 정밀하게 다듬습니다.

RTAB-Map은 로봇이 이동하면서 일정 간격마다 "키프레임"(대표 장면 스냅샷)을 저장하고, 이 키프레임들을 점과 점을 잇는 선(링크)으로 연결한 그래프를 만듭니다. 이 그래프가 곧 지도의 뼈대입니다.

키프레임을 처음 저장할 때 만들어지는 "이웃 링크(neighbor link)"는 주로 오도메트리(바퀴 추정값) 기반이라 약간 부정확할 수 있습니다. NeighborLinkRefining을 켜면, 이 링크를 라이다 스캔 데이터로 한 번 더 ICP 정렬해서 "진짜 정확한 상대 위치"로 보정합니다.

  • 실생활 비유: 사진첩에 사진을 순서대로 끼워 넣은 다음, 인접한 두 장을 나란히 놓고 "어, 이 두 사진이 실제로는 살짝 겹치네?"라며 위치를 미세 조정하는 것과 같습니다.
  • 끄면 어떻게 되나? 오도메트리 오차가 그대로 그래프에 남아서, 누적되면 지도가 점점 휘어지거나 벽이 두 겹으로 겹쳐 보이는 현상(고스트 벽)이 생기기 쉽습니다.

Grid/RangeMin: 0.2 — "내 몸은 장애물로 치지 않기"

한 줄 요약 라이다로부터 0.2m(20cm) 이내의 점은 점유 격자 지도를 만들 때 무시합니다.

라이다는 360도로 빛(레이저)을 쏘는데, 로봇 자신의 몸체(범퍼, 케이블, 카메라 마운트 등)에 너무 가까이 붙어 있으면 그 반사광도 잡혀서 "바로 코앞에 벽이 있다"고 잘못 인식할 수 있습니다.

  • 왜 0.2m인가? R2 로봇 본체의 반경보다 약간 큰 값으로 설정해서, 라이다 바로 앞의 로봇 부품들이 지도에 "유령 장애물"로 찍히지 않도록 안전 여유를 둔 것입니다.
  • 실생활 비유: 사진을 찍을 때 카메라 렌즈에 내 손가락이 살짝 걸리면 그 부분은 사진(지도)에서 무시하고 넘어가는 것과 비슷합니다.
  • 값이 너무 작으면? 로봇 자신의 일부가 지도에 장애물로 찍혀서, 정작 그 위치를 영원히 못 지나가는 곳으로 착각할 수 있습니다.
  • 값이 너무 크면? 진짜 가까이 있는 작은 장애물(발끝의 슬리퍼 등)을 놓쳐서 충돌 위험이 커집니다.

Optimizer/GravitySigma: 0 — "중력 센서값은 믿지 않기"

한 줄 요약 그래프를 최적화(보정)할 때 IMU(관성 센서)가 측정한 "중력 방향" 정보를 제약 조건으로 쓰지 않도록 끕니다.

IMU는 가속도계를 이용해 "아래쪽이 어디인지"(중력 방향)를 알 수 있습니다. 드론처럼 3차원으로 기울어지는 로봇은 이 정보가 "내가 똑바로 서 있나, 기울었나"를 판단하는 데 중요한 단서가 됩니다.

하지만 R2는 위에서 설명한 Force3DoF처럼 애초에 기울어짐을 고려하지 않는 2D 로봇입니다. GravitySigma: 0은 "이 IMU 중력 정보를 그래프 최적화에 아예 반영하지 마라"는 뜻으로, 불필요한(또는 바닥 진동으로 인해 오히려 잘못된) 제약을 제거합니다.

  • Sigma(시그마)란? 통계에서 "표준편차", 즉 측정값의 불확실성(흔들림 정도)을 나타내는 기호입니다. 값이 0이라는 것은 "이 측정값의 신뢰도를 0으로 취급해서 사실상 무시하라"는 의미로 흔히 쓰입니다.
  • 실생활 비유: 평지에서 자전거를 타는데 굳이 고도계(높이 측정기) 값을 참고해서 방향을 정하지 않는 것과 같습니다 — 어차피 평평한 길이니 필요 없는 정보입니다.

subscribe_rgbd / subscribe_scan: True — "두 귀로 동시에 듣기"

한 줄 요약 RTAB-Map 노드가 카메라 데이터(rgbd_image)와 라이다 데이터(scan)를 둘 다 구독(수신 대기)하도록 켜는 스위치입니다.

RTAB-Map은 원래 카메라만으로도, 라이다만으로도 SLAM을 할 수 있는 다재다능한 도구입니다. 이 두 옵션은 "둘 중 어떤 센서를 켤지" 선택하는 스위치이며, 이 프로젝트에서는 둘 다 켜서 함께 사용(센서 퓨전)합니다.

옵션끄면 (False)켜면 (True, 이 프로젝트 설정)
subscribe_rgbd 카메라 영상 무시, 라이다 점만으로 SLAM 카메라 영상(색상+깊이)도 함께 분석 — 시각적 특징으로 장소 구분
subscribe_scan 라이다 무시, 카메라만으로 SLAM 라이다의 정밀한 거리 측정도 함께 사용 — 벽 경계가 더 정확
  • 왜 둘 다 켜나? 라이다는 "거리"는 정확하지만 "이 복도가 저 복도와 뭐가 다른지" 구분을 잘 못합니다 (둘 다 그냥 평행한 벽 두 개로 보임). 반대로 카메라는 "여기는 빨간 문이 있는 곳"처럼 시각적 특징은 잘 잡지만 거리 측정은 라이다보다 부정확합니다. 둘을 합치면 서로의 약점을 보완합니다.
  • 실생활 비유: 눈(카메라, 풍경을 기억)과 손(라이다, 거리를 더듬어 측정)을 동시에 써서 길을 찾는 것과 비슷합니다. 눈을 감아도 손으로 벽 거리를 알 수 있고, 손이 닿지 않는 곳도 눈으로 구분할 수 있습니다.

TF 트리 — 좌표 변환 체인

ROS2에서 "로봇이 지도 위 어디에 있는가"는 TF(Transform) 프레임 체인으로 결정됩니다.

map RTAB-Map이 발행 (SLAM 보정값) └── odom 바퀴 엔코더가 발행 (누적 오차 있음) └── base_footprint 로봇 바닥면 └── base_link 로봇 본체 중심 ├── laser 라이다 (static TF) ├── laser_link 라이다 링크 ├── camera_link 카메라 본체 │ ├── camera_color_frame │ │ └── camera_color_optical_frame │ └── camera_depth_frame │ └── camera_depth_optical_frame ├── imu_link 관성 센서 ├── front_left_wheel 앞바퀴 (좌) ├── front_right_wheel 앞바퀴 (우) ├── back_left_wheel 뒷바퀴 (좌) └── back_right_wheel 뒷바퀴 (우)

각 변환의 역할

변환발행자의미
map → odom RTAB-Map SLAM이 계산한 보정값. 바퀴 오도메트리의 누적 오차를 보상
odom → base_footprint yahboomcar_bringup (엔코더) 바퀴 회전으로 추정한 로봇 이동량. 미끄러짐 등으로 점점 틀어짐
base_link → laser static_transform_publisher 라이다가 로봇 위 어디에 장착되어 있는지 (고정값)
base_link → camera_link robot_state_publisher (URDF) 카메라 장착 위치 (URDF 모델에서 정의)
왜 map → odom → base_footprint 2단계인가? 바퀴 엔코더(odom → base_footprint)는 빠르게 갱신되지만 누적 오차가 생깁니다. SLAM(map → odom)은 느리지만 루프 클로저로 오차를 한꺼번에 보정합니다. 이 2단계 구조 덕분에 빠른 엔코더 + 정확한 SLAM을 동시에 활용할 수 있습니다.

지도 관리 체계

map_registry.py가 여러 장소의 지도를 체계적으로 관리합니다.

디렉터리 구조

~/maps/
├── databases/          # RTAB-Map DB 파일들 (각 장소별)
│   ├── loc_p37.5665_p126.9780.db    # 위치 좌표 기반 자동 이름
│   └── ...
├── exported/           # Nav2용으로 내보낸 2D 지도
│   └── loc_xxx/
│       ├── nav_map.yaml
│       └── nav_map.pgm
├── patrol/             # 순찰 웨이포인트
├── geofence/           # 울타리 영역 정보
└── registry.json       # 모든 지도 메타데이터 (위치, 이름, 경로)

~/.ros/rtabmap.db  →  심볼릭 링크 → 현재 활성 지도 DB

운영 워크플로

새 지도 생성

# start_rtabmap_new.sh 실행 시 내부 동작:
python3 map_registry.py new          # IP 기반 위치 추정 → DB 파일 생성
  → 위치 좌표로 map_id 자동 생성 (예: loc_p37.5665_p126.9780)
  → ~/maps/databases/{map_id}.db 경로 등록
  → ~/.ros/rtabmap.db 심볼릭 링크 갱신
  → DELETE_DB_ON_START=1 설정

bash rtabmap_mapping_core.sh new
  → 카메라 시작 (astro_pro_plus.launch.xml)
  → 카메라 토픽 대기 (최대 25초)
  → rtabmap_managed.launch.py 실행 (delete_db:=true)

기존 지도 이어서 매핑

# start_rtabmap_continue.sh 실행 시:
python3 map_registry.py continue     # 활성 지도 DB 경로 조회
  → DELETE_DB_ON_START=0 설정

bash rtabmap_mapping_core.sh continue
  → rtabmap_managed.launch.py 실행 (delete_db:=false)

지도 이름 지정

# 자동 생성된 좌표 ID를 사람이 읽기 쉬운 이름으로
python3 map_registry.py rename '집'
python3 map_registry.py rename '회사'

순찰 워크플로 (Nav2 연동)

start_patrol.sh는 매핑된 지도 위에서 자율 순찰을 수행합니다. 4단계로 진행됩니다.

[1/4] 카메라 시작 │ Orbbec Astra Pro Plus → /camera/* 토픽 대기 ▼ [2/4] RTAB-Map 로컬라이제이션 │ rtabmap_localization_managed.launch.py │ → 기존 DB 위에서 현재 위치만 추적 (지도 추가 안 함) │ │ [2b] 2D 지도 내보내기 (아직 없으면) │ export_rtabmap_map.sh │ → /map 토픽에서 nav_map.yaml + nav_map.pgm 저장 │ → map_saver_cli 사용 ▼ [3/4] Nav2 시작 │ nav2_bringup → bringup_launch.py │ → map_server: nav_map.yaml 로드 (lifecycle: active 대기) │ → AMCL: 라이다 기반 위치 추정 (파티클 필터) │ → bt_navigator: 행동 트리 내비게이션 │ │ [3b] AMCL 초기 위치 주입 │ publish_initial_pose.py │ → RTAB-Map의 현재 추정 위치를 AMCL에 전달 ▼ [4/4] 순찰 루프 │ geofence_patrol.py --loop --margin 1.0 --waypoint-inset 0.8 │ → 지도 PGM에서 free cell 분석 → 안전한 웨이포인트 생성 │ → Nav2의 BasicNavigator로 웨이포인트 순회 │ → 장애물 회피: Nav2 DWB 플래너 + 라이다 costmap ▼ Ctrl+C로 중지
웨이포인트 생성 원리 geofence_patrol.py의 waypoints_from_map_yaml() 함수는 내보낸 PGM 이미지를 직접 읽어서, 각 셀의 밝기값이 250 이상(확실히 빈 공간)이고 주변 반경 내에도 장애물이 없는 셀만 웨이포인트 후보로 선택합니다. 지도 가장자리의 미탐사 영역(회색)이나 벽 근처는 자동으로 제외됩니다.

부록: OccupancyGrid 셀 값의 의미

RTAB-Map이 발행하는 /map 토픽은 nav_msgs/OccupancyGrid 메시지입니다. 각 셀은 0~100 또는 -1 값을 가집니다.

셀 값RViz 색상의미Nav2 해석
0 흰색 확실히 빈 공간 (로봇이 지나갈 수 있음) 이동 가능 영역
100 검은색 확실히 장애물 (벽, 가구 등) 이동 불가 + 주변에 inflation 비용 부여
-1 회색 미탐사 영역 (아직 센서가 닿지 않은 곳) Nav2 설정에 따라 이동 허용 또는 차단

PGM 파일로 내보낼 때

map_saver_cli로 내보내면 .yaml + .pgm 파일 쌍이 생성됩니다.

  • PGM 픽셀값 254~255: 빈 공간 (free, 밝음)
  • PGM 픽셀값 0~1: 장애물 (occupied, 어두움)
  • PGM 픽셀값 205: 미탐사 (unknown, 중간 밝기)

YAML 파일에는 해상도(resolution: 0.05), 원점 좌표(origin), 점유 판정 임계값(occupied_thresh: 0.65, free_thresh: 0.25)이 기록됩니다.
