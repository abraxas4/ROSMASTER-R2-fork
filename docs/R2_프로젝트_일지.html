1. 왜 이 프로젝트를 시작했나

최종 목표 집 안에서 ROSMASTER R2로 SLAM(동시 위치 추정·지도 작성)을 하고, 나중에 자율 주행 내비게이션까지 가는 것.

Yahboom에서 ROSMASTER R2를 구매했습니다. 공식 튜토리얼과 Google Drive의 5.Code 패키지가 있지만, 그대로 쓰면:

  • 코드 수정이 로봇 본체에만 남고 버전 관리가 안 됨
  • Yahboom 원본(upstream)에 실수로 commit할 위험이 있음
  • PC에서 SSH로 원격 개발하면서 로봇에는 git pull만 받고 싶음
  • 불필요한 프로세스는 최소화 — "the best part is no part"

그래서 GitHub에 자기만의 fork(abraxas4/ROSMASTER-R2-fork)를 만들고, ROS2 코드를 그 안에 넣어 관리하기로 했습니다. Yahboom 원본은 건드리지 않고, 로봇에 이미 깔려 있던 ~/yahboomcar_ros2_ws도 당분간 삭제하지 않고 병행 사용했습니다.

3. Fork란 무엇인가 — Upstream과의 관계

한 줄 요약 Fork = Yahboom 원본 저장소의 내 계정에 복사본. 내 복사본만 수정·commit하고, 원본에는 절대 push하지 않음.
Yahboom 원본 (upstream) 내 Fork (origin) YahboomTechnology/ROSMASTER-R2 → abraxas4/ROSMASTER-R2-fork │ │ │ (읽기만, merge 선택) │ commit & push └────────────────────────────────────┘ │ git clone / git pull │ 로봇 Jetson ~/ROSMASTER-R2-fork

Google Drive 5.Code는 왜 복사하나?

Yahboom이 GitHub에는 튜토리얼 PDF만 올리고, 실제 ROS2 패키지 소스는 Google Drive 5.Code에 따로 둡니다. 이걸 fork 안의 code/yahboomcar_ros2_ws/src/에 넣으면:

  • 22개 ROS2 패키지가 git으로 관리됨
  • PC에서 수정 → push → 로버에서 pull
  • Upstream Yahboom에는 영향 없음

scripts/download_code.sh로 Google Drive에서 자동 다운로드·설치할 수 있게 만들었고, 큰 파일 ORBvoc.txt는 git 제한 때문에 분할 저장(split_large_files.sh / restore_large_files.sh)했습니다.

4. 폴더 구조 — 두 개의 워크스페이스

로봇에는 두 군데에 ROS 관련 코드가 있습니다. 처음엔 헷갈리기 쉬운데, 역할이 다릅니다.

경로역할크기(대략)삭제?
~/yahboomcar_ros2_ws Yahboom 출하 시 설치된 원본. library_ws(라이다·SLAM·카메라 드라이버) + yahboomcar_ws(bringup·nav) 포함. 지금 매핑에 실제로 사용 중. ~2.9GB 아직 NO
~/ROSMASTER-R2-fork 내가 만든 git 관리 fork. 코드 수정·스크립트·문서. Docker로 일부 패키지 빌드. ~1.9GB 메인 개발 repo
~/ROSMASTER-R2-fork/ ├── code/yahboomcar_ros2_ws/ ← fork 안 ROS 패키지 (git 관리) │ └── src/ ← Google Drive 5.Code 내용 ├── scripts/ ← 진단·매핑·Docker·바탕화면 설치 스크립트 ├── README.md, DEVELOPMENT.md └── (Yahboom PDF 튜토리얼) ~/yahboomcar_ros2_ws/ ← Yahboom 원본 (아직 런타임에 필요) ├── software/library_ws/ ← ydlidar, slam_gmapping, astra_camera, robot_localization └── yahboomcar_ws/ ← yahboomcar_nav, bringup, launch 파일
왜 원본을 아직 안 지우나? library_ws는 Jetson 호스트용 GLIBC 2.35로 빌드되어 있어서, Foxy Docker 안에서는 실행이 안 됩니다. 매핑·센서는 이 바이너리를 native Humble로 실행합니다. fork를 Humble에서 완전히 빌드할 때까지 원본이 필요합니다.

5. 단계별로 한 일 (타임라인)

  1. ① Git 개발 구조 만들기

    Fork repo 생성, DEVELOPMENT.md 작성, scripts/update_code.sh로 로버에서 git pull만 받도록 설계. 사용자 홈 경로 자동 감지(~abraxas4 / ~jetson).

  2. ② Google Drive Code 설치

    download_code.sh로 5.Code → code/yahboomcar_ros2_ws/src/. 22개 패키지 중 git에 포함.

  3. ③ Docker로 colcon build

    이미지 ros-foxy:4.0.7R2 사용. 18개 패키지 빌드 성공. 4개 스킵: KCFTracker, mediapipe, visual, slam (의존성·호환 문제).

  4. ④ 빌드 오류 수정 (fork 코드)

    yahboomcar_base_node: 안 쓰는 turtlesim include 제거. yahboomcar_KCFTracker: OpenCV 4.2 맞춤. CMakeLists 정리.

  5. ⑤ 불필요 자동실행 끄기

    Yahboom 폰앱 rosmaster_main.py autostart 비활성화 — 이게 /dev/myserial을 잡아서 ROS 드라이버와 충돌함. disable_yahboom_autostart.sh 작성.

  6. ⑥ 하드웨어 진단 스크립트

    diagnose_robot.sh: 장치·워크스페이스·선택적 ROS 토픽 확인. 매핑 전에 Phase 1 체크.

  7. ⑦ Native Humble로 전환 (결정적 전환점)

    Docker에서 라이다·SLAM 실행 시 GLIBC 오류 발견 → native_ros_setup.bash + start_mapping.sh(gmapping) 작성. 센서·매핑은 호스트 Humble만 사용.

  8. ⑧ Orbbec 카메라 연동

    start_camera.sh, diagnose_camera.sh. astra_camera astro_pro_plus.launch.xml로 RGB ~25–30Hz, depth, point cloud 확인.

  9. ⑨ RTAB-Map 융합 SLAM

    카메라+라이다 RTAB-Map: start_rtabmap_mapping.sh, diagnose_rtabmap.sh. 맵 DB: ~/.ros/rtabmap.db.

  10. ⑩ 실시간 맵 화면 + 바탕화면 아이콘

    view_rtabmap_viz.sh, scripts/desktop/*.desktop 4개, install_desktop_shortcuts.sh. R2 모니터에서 더블클릭으로 매핑·보기·중지.

  11. ⑪ 중지·비상 정리 강화

    stop_robot_stack.sh 개선 (rtabmap_viz, IMU, EKF까지 종료). emergency_stop.sh 추가. 좀비 프로세스·연속 비프음 대응.

  12. ⑫ 실제 매핑 테스트 성공

    배터리 충전 중 부팅, 비프음 없음. 바탕화면에서 매핑 시작 → 실시간 맵 화면 확인. 중지 아이콘으로 종료 (카메라 LED 대기 상태 확인).

6. 실제로 만난 문제와 해결

SLAM이 "한 번에" 된 것이 아닙니다. 아래 문제들을 하나씩 해결했습니다.

# 증상 / 문제 원인 해결
1 Google Drive Code를 fork에 어떻게 넣지? upstream에 commit하고 싶지 않음 Yahboom 구조상 코드가 Drive와 GitHub가 분리됨 code/yahboomcar_ros2_ws/src/에 복사 후 내 fork에만 commit. DEVELOPMENT.md에 워크플로 문서화
2 Docker colcon build 실패 turtlesim 미사용 include, OpenCV 버전 불일치, 선택 패키지 의존성 base_node CMake 수정, OpenCV 4.2, 4개 패키지 --packages-skip
3 Docker 안에서 라이다·gmapping 실행 불가 library_ws가 호스트 GLIBC 2.35용으로 빌드됨. Foxy 컨테이너 GLIBC가 다름 센서·SLAM은 Native Humble로 실행. Docker는 코드 빌드용으로만
4 /dev/myserial 열기 실패, 매핑 시작 안 됨 Yahboom 폰앱 rosmaster_main.py가 시리얼 독점 autostart 비활성화 + 매핑 전 stop_robot_stack.sh로 프로세스 종료
5 Ctrl+C 후에도 모터·노드가 살아 있음, 비프음 계속 launch가 여러 자식 프로세스 생성, 정리 안 됨 stop_robot_stack.sh: pkill 목록 확장 + Rosmaster_Lib로 set_beep(0), reset_car_state()
6 배터리 방전/보드 미응답 시 battery=0.0, version=-1, 비프음 안 꺼짐 모터보드 하드웨어 알람. 소프트웨어만으로 한계 배터리 충전 + 메인 스위치 OFF/ON. 부팅 중 비프음 = 하드웨어 알람
7 SSH로 매핑해도 R2 모니터에 맵 안 보임 SSH 세션에 DISPLAY 없음 view_rtabmap_viz.sh + 바탕화면 아이콘 (DISPLAY=:0)
8 diagnose_rtabmap.sh가 중간에 죽음 set -e + timeout ros2 topic hz exit code 124 timeout 실패를 || true로 처리, echo 폴백
9 중지 아이콘 눌렀는데 카메라 LED 빠르게 깜빡임 rtabmap_viz 프로세스가 남음. 중지 스크립트가 launch 파일만 kill pkill rtabmap_viz 등 추가. 맵 보기 창도 함께 닫기
10 중지 후 카메라 LED 연한 빨강·녹색 USB 전원은 남고 소프트웨어만 종료된 idle 상태. 드라이버 일시 꼬임 시 /dev/video 사라짐 정상 대기로 판단. 카메라 30Hz 테스트 OK. 필요 시 modprobe uvcvideo 또는 전원 OFF
11 집 WiFi SSH 끊김 로봇이 ROSMASTER 핫스팟에 자동 연결 NetworkManager에서 핫스팟 autoconnect off, 집 WiFi 우선순위 up

7. 핵심 결정: Docker vs Native Humble

이 프로젝트에서 가장 중요한 기술적 깨달음입니다.

용도환경이유
코드 빌드 (colcon build) Docker Foxy 4.0.7R2 Yahboom이 이 이미지 기준으로 패키지 제공. PC/로버 동일 환경
라이다, 카메라, SLAM, 매핑 Native Humble (호스트) library_ws 바이너리가 호스트용. Docker에서 실행 시 GLIBC 오류

native_ros_setup.bash가 하는 일:

source /opt/ros/humble/setup.bash
source ~/yahboomcar_ros2_ws/software/library_ws/install/setup.bash
source ~/yahboomcar_ros2_ws/yahboomcar_ws/install/setup.bash
export ROS_DOMAIN_ID=28
export ROBOT_TYPE=r2
export RPLIDAR_TYPE=4ROS
export CAMERA_TYPE=astraplus
교훈 "Docker로 다 하면 깔끔하겠지?" → R2에서는 센서 드라이버가 이미 호스트용으로 빌드되어 있어서 안 됨. 빌드와 런타임을 분리한 것이 정답이었습니다.

8. SLAM 매핑이 동작하게 된 방법

방식 A: 라이다만 (gmapping)

가볍고 빠른 테스트용.

bash ~/ROSMASTER-R2-fork/scripts/start_mapping.sh
# 내부: ros2 launch yahboomcar_nav map_gmapping_4ros_launch.py

방식 B: 카메라 + 라이다 융합 (RTAB-Map) — 실제로 성공한 방식

더 풍부한 지도. 집 실내 SLAM에 적합.

  1. stop_robot_stack.sh — 기존 프로세스·시리얼 점유 해제
  2. Orbbec 카메라 launch — /camera/color/image_raw 토픽 대기 (~25초)
  3. map_rtabmap_launch.py — RGB-D + /scan 융합
  4. 로봇을 천천히 움직이며 지도 생성
  5. 별도 창에서 view_rtabmap_viz.sh — 실시간 2D/3D 맵 확인

저장 위치:

  • RTAB-Map DB: ~/.ros/rtabmap.db (약 2.9MB — 테스트 매핑 기준)
  • 보내기용 ~/maps/: 아직 yaml/pgm보내기 전 (다음 단계)

정상 동작 시 확인되는 ROS 토픽 (예시)

  • /scan — 라이다, ~10Hz
  • /camera/color/image_raw — RGB, ~25–30Hz
  • /camera/depth/image_raw, /camera/depth/points
  • /imu/data_raw, /odom_raw, /vel_raw

9. 바탕화면 아이콘과 실시간 맵 보기

R2 본체 모니터(로컬 디스플레이 :0)에서 더블클릭으로 실행합니다. SSH만으로는 GUI가 안 뜹니다.

아이콘 이름하는 일
R2 매핑 시작 (카메라+라이다)start_rtabmap_mapping.sh
R2 맵 실시간 보기view_rtabmap_viz.sh → RTAB-Map viz 창
R2 매핑 중지stop_robot_stack.sh
R2 라이다만 매핑 (gmapping)start_mapping.sh

설치 방법 (로버에서, git pull 후):

bash ~/ROSMASTER-R2-fork/scripts/install_desktop_shortcuts.sh

매핑 시작 → 중지 올바른 순서

  1. R2 매핑 시작 더블클릭
  2. R2 맵 실시간 보기 더블클릭 (별도 터미널/창)
  3. 로봇 천천히 이동하며 맵 확인
  4. R2 매핑 중지 더블클릭 (또는 매핑 터미널에서 Ctrl+C)
  5. 맵 보기 창도 닫기
카메라 LED 참고
  • 전원 OFF → LED 꺼짐
  • 전원 ON, 앱 안 씀 → 연한 빨강·녹색 (idle, 정상)
  • 매핑/카메라 실행 중 → 밝고 빠른 깜빡임

예전 아이콘 R2_Mapping.desktop, R2_RViz.desktop은 ~/yahboomcar_ros2_ws 경로를 가리킵니다. 새 아이콘 4개를 사용하세요.

11. 지금 상태와 앞으로 할 일

달성한 것
  • Git fork로 코드·스크립트 버전 관리
  • PC → push → 로버 pull 워크플로
  • Native Humble에서 4ROS 라이다 + Orbbec 카메라 + RTAB-Map 매핑 동작
  • R2 바탕화면에서 매핑·실시간 맵·중지
  • 진단·중지·비상 스크립트로 운영 가능

아직 안 한 것 (다음 단계)

  • RTAB-Map 지도를 ~/maps/에 yaml/pgm으로 보내기 (rtabmap-export)
  • 보낸 지도로 내비게이션 (amcl + nav2 또는 Yahboom nav launch)
  • fork의 library_ws 의존 제거 — Humble에서 전체 빌드
